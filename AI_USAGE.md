## Development Dynamic
This project was built from scratch utilizing a strictly defined collaboration model:
- **The User** acted as the **Product Manager and Senior Developer**, defining the architecture, writing the business requirements, handling anomaly edge-cases, and guiding the implementation.
- **Antigravity AI (Gemini)** acted as the **Junior Engineer**, executing the code, scaffolding the boilerplate, and implementing the requested features under the user's guidance.

## How AI Was Used

### 1. Boilerplate Acceleration
The FastAPI router structure, SQLAlchemy model definitions, and Pydantic schema boilerplate were generated with AI assistance and then verified against the docs. This saved approximately 40% of the initial scaffolding time.

### 2. Min-Cash-Flow Algorithm
The `simplify_debts` function in `balances.py` was initially generated by AI. The algorithm logic (greedy creditor/debtor matching) was verified by hand-tracing through a 4-person example before being accepted.

### 3. CSV Parser Heuristics
The anomaly detection rules in `import_parser.py` — especially the settlement keyword list and percentage sum check — were drafted with AI assistance, then reviewed and extended with domain-specific knowledge (e.g. adding "paid back", "reimbursed" as settlement signals).

### 4. React Component Scaffolding
The `ImportCSVModal.tsx` component structure was AI-generated and then heavily customised for the multi-step UX and per-anomaly decision UI.

## Cases Where AI Was Wrong

### ❌ Error 1: Hard Delete for Group Members

**AI's initial suggestion**: When removing a user from a group, use `db.delete(db_member)` and `db.commit()` — a permanent hard delete.

**Why it was wrong**: This destroys the historical membership record. Once Meera's `group_members` row is deleted, it becomes impossible to know she was in the group at all — the time-aware balance calculation has nothing to filter on. Meera's historical expenses would simply be allocated to nobody (or cause SQL errors on the `member_ids` filter).

**What I did instead**: Changed to a soft delete: `db_member.left_at = datetime.utcnow()`. The row stays, just with a `left_at` timestamp set. The balance query then filters `expenses.created_at <= left_at`.

---

### ❌ Error 2: Balance Calculation Using `group.members` Relationship

**AI's initial code**:
```python
member_ids = [m.id for m in group.members]
```
Followed by `GROUP BY user_id` aggregate queries filtered by `member_ids`.

**Why it was wrong**: `group.members` returns only the `User` objects linked through the ORM relationship — which at the time was a direct join without filtering on `left_at`. So an ex-member like Meera who had been soft-deleted (her `left_at` was set) would either still appear (because the relationship wasn't filtering) or disappear completely (losing her historical data). Either way, the balance would be wrong.

**What I did instead**: Rewrote `get_group_member_balances()` to first fetch all `GroupMember` rows directly (including ex-members), build a time-window map `{user_id: (joined_at, left_at)}`, then for each **active** member (those with `left_at == None`), run per-member queries with date filters on `expenses.created_at`. This correctly excludes expenses outside the membership window while only showing active members in the balance output.

### ❌ Error 3: Currency Migration as an Inline String in PowerShell

**AI's initial approach**: Running the migration as a one-line Python `-c` string in PowerShell:
```powershell
.venv\Scripts\python.exe -c "... conn.execute(sa.text(\"ALTER TABLE expenses ADD COLUMN currency VARCHAR DEFAULT 'INR' NOT NULL\")) ..."
```

**Why it was wrong**: PowerShell has different quoting rules than Bash. The nested double-quotes inside the `-c` string caused `SyntaxError: unterminated string literal` because the single quotes inside the SQL string were being interpreted as string delimiters by PowerShell's command parser.

**What I did instead**: Extracted the migration into a standalone `migrate.py` file and ran it with `python migrate.py`. This avoids all shell quoting issues, is more readable, and is idempotent (checks column existence before ALTER).

## Prompting Notes

- "balances should only include expenses that occurred while the member was active — use joined_at and left_at timestamps". The initial output had the right intent but the wrong implementation.

- "Do not silently fix anything — every change must be surfaced as an anomaly item that the user reviews and accepts or rejects." This constraint was critical to produce the right two-phase design.

- "Do not store a single payer_id on the expenses table. Create an `expense_payers` table to support split payments, and validate that the sum of all payer contributions equals the total expense amount before executing db writes."

- "Ensure WebSocket connections are authorized. Since standard client WebSocket APIs do not support custom authorization headers, pass the JWT access token in the query string (e.g. `?token=JWT_TOKEN`) and decode it during the connection handshake."

- "To prevent floating-point rounding errors on divisions, map all monetary values to PostgreSQL `NUMERIC(10,2)` and use Python's `Decimal` module for all calculation algorithms, rounding to exactly 2 decimal places."

- "Add a separate, persistent group chat WebSocket (`/ws/groups/{group_id}/chat`) that logs conversation records to a `group_messages` table, coupled with a REST API to fetch initial messages history when the group view loads."
